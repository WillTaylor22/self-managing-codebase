During a `git merge origin/main` with a single conflicted hunk, `git checkout --theirs <path>` replaces the **entire file** with the incoming version — it does NOT just resolve the conflicted region. Any non-conflicting edits I had committed earlier on the branch in unrelated parts of the same file get silently reverted. Hit during PR #29 review-feedback round 1: my install-section fix conflicted with `a028eb3`, my `Bootstrapping is human-only` line on a far-away bullet was clean — but `--theirs` blew it away too. Always run `git diff origin/<base> -- <path>` after a `--theirs` resolution to verify the surviving diff is what you intended; re-apply any unrelated edits explicitly. Better default: hand-edit conflict markers when the file has unrelated changes elsewhere, and reserve `--theirs/--ours` for fully replacing files (binary assets, generated code).
